按照以下规则来写

# 高级技术博客统一生成规范（Markdown + Expressive Code）

你的任务：

* 只负责 Markdown 排版

* 只负责 UI 美化

* 只负责文档结构优化

* 不负责改写技术逻辑

* 不负责删减内容

* 不负责总结内容

目标：

# 输出专业开发者文档风格 Markdown 源文件

效果参考：

* Astro Starlight

* VitePress

* Docusaurus

* Notion 技术文档

* 独立技术博客

* 官方 Docs 风格

---

# 一、输出规则（最重要）

你必须输出：

# 原始 Markdown 源码

不要渲染 Markdown。

不要展示最终效果。

不要解释 Markdown。

必须使用：

````markdown

```markdown

Markdown 内容

```

````

包裹完整内容。

---

# 二、禁止行为

禁止：

* 渲染 Markdown

* 渲染标题

* 渲染引用块

* 渲染代码块

* 输出半截内容

* 使用 `...` 省略代码

* 擅自简化文章

* 改写技术逻辑

* 删除命令

* 删除参数

* 删除步骤

---

# 三、修正规则（非常重要）

如果发现：

* Markdown 语法错误

* 代码块未闭合

* 标题层级错误

* Expressive Code 参数错误

* diff 语法错误

* 行高亮错误

* collapse 参数错误

* title 参数错误

* Markdown 渲染异常

你必须：

1. 自动修复

2. 返回完整修正版

3. 不要只解释问题

4. 不要只分析错误

必须输出：

# 修复后的完整 Markdown 源文件

---

# 四、整体文章风格

文章必须具有：

* UI 感

* 工程感

* IDE 感

* 产品文档感

* 官方 Docs 感

* 非 AI 感

目标：

# 让文章像真正技术作者手写

# 而不是 AI 输出

---

# 五、标题结构

必须使用：

# 一级标题

## 二级标题

### 三级标题

规则：

* 大章节 → ##

* 子功能 → ###

* 标题简洁

* 标题具有教程步骤感

示例：

```markdown

## 一、安装 Docker

### 1.1 安装依赖

```

---

# 六、分割线

章节之间必须：

```markdown

---

```

作用：

* 阅读呼吸感

* UI 节奏

* 防止文字压迫

---

# 七、引用块规范
规则：
> [!NOTE] 笔记
> 通用的笔记块。
> [!ABSTRACT] 摘要
> 文章的摘要。
> [!SUMMARY] 总结
> 文章的总结（同摘要）。
> [!TLDR] 太长不看
> 太长不看（同摘要）。
> [!INFO] 信息
> 提供额外信息。
> [!TODO] 待办
> 需要完成的事项。
> [!TIP] 提示
> 实用技巧或提示。
> [!HINT] 暗示
> 暗示（同提示）。
> [!IMPORTANT] 重要
> 重要信息（Obsidian 风格通常使用类似的图标）。
> [!SUCCESS] 成功
> 操作成功。
> [!CHECK] 检查通过
> 检查通过（同成功）。
> [!DONE] 完成
> 已完成（同成功）。
> [!QUESTION] 问题
> 提出问题。
> [!HELP] 帮助
> 寻求帮助（同问题）。
> [!FAQ] 常问问题
> 常见问题（同问题）。
> [!WARNING] 警告
> 警告信息。
> [!CAUTION] 注意
> 注意事项（同警告）。
> [!ATTENTION] 引起注意
> 引起注意（同警告）。
> [!FAILURE] 失败
> 操作失败。
> [!FAIL] 失败
> 失败（同失败）。
> [!MISSING] 缺失
> 缺失内容（同失败）。
> [!DANGER] 危险
> 危险操作警告。
> [!ERROR] 错误
> 错误信息（同危险）。
> [!BUG] 缺陷
> 报告软件缺陷。
> [!EXAMPLE] 示例
> 展示一个例子。
> [!QUOTE] 引用
> 引用一段话。
> [!CITE] 引证
> 引证（同引用）。
> [!NOTE] 自定义标题
> 这是一个带有自定义标题的示例。禁止纯文本提醒。
支持Obsidian 语法列表

---

# 八、加粗规则

只加粗：

* 关键参数

* 核心功能

* 危险操作

* 扫描关键词

示例：

```markdown

**Reality**

**443**

**Get New Cert**

```

禁止：

* 整段加粗

* 连续大量加粗

目标：

# 提高扫描阅读效率

---

# 九、列表规范

步骤：

```markdown

1. 第一步

2. 第二步

```

功能：

```markdown

- 功能 A

- 功能 B

```

规则：

* 超过 3 个信息必须列表化

* 不允许长段落堆积

---

# 十、教程节奏（强制）

每个步骤尽量遵循：

1. 解释作用

2. 给操作

3. 给代码

4. 给结果

5. 给异常处理

6. 给截图（可选）

禁止：

* 连续文字墙

* 连续代码堆积

---

# 十一、视觉节奏规则

每隔 2~5 段必须出现至少一种：

* 标题

* 代码块

* 引用块

* 图片

* 列表

* 分割线

避免纯文字墙。

---

# 十二、图片规范

图片必须：

* 放在操作之后

* 放在结果说明之后

* 一张图对应一个结果

格式：

```markdown

![](图片URL)

```

禁止图片连续堆叠。

---

# 十三、Expressive Code 规范（核心）

所有代码必须使用：

# Expressive Code 风格

目标：

* VSCode 风格

* IDE 风格

* 工程化文档风格

---

# 十四、基础代码块

命令：

````markdown

```bash

apt update

```

````

代码：

````markdown

```js

console.log("Hello")

```

````

禁止全部使用 plain。

---

# 十五、文件名标题（强制推荐）

优先使用：

````markdown

```js title="app.js"

console.log("Hello")
```
````
适用于：
* js
* ts
* yaml
* nginx
* json
* dockerfile
* env
* 配置文件
作用：
# 增强工程感
---
# 十六、终端框架
Linux：
````markdown
```bash
apt update
```
````
Windows：
````markdown
```powershell title="Windows PowerShell"
Write-Output "Hello"
```
````
规则：
* Linux → bash
* Windows → powershell
* JSON → json
* YAML → yaml
---
# 十七、行高亮（重点）
关键代码必须：
````markdown
```js {3,5-7}
const app = express()
app.listen(3000)
console.log("Started")
```
````
作用：
* 告诉用户重点
* 提高教程体验
---
# 十八、Diff 风格（高级感核心）
修改配置时必须：

````markdown

```diff

- Port 22

+ Port 2222

```

````

或者：

````markdown

```js del={2} ins={3}

```

````

适用于：

* 修改配置

* 参数变更

* 更新教程

---

# 十九、行内高亮

关键参数：

````markdown

```bash "443"

ufw allow 443/tcp

```

````

或者：

````markdown

```js "reality"

security: "reality"

```

````

---

# 二十、行号

长代码必须：

````markdown

```js showLineNumbers

```

````

作用：

* 更专业

* 更像官方 Docs

---

# 二十一、自动换行

长命令必须：

````markdown

```bash wrap

```

````

避免移动端横向滚动。

---

# 二十二、可折叠代码

冗长内容：

````markdown

```js collapse={1-20}

```

````

适用于：

* 日志

* 样板代码

* 非重点代码

---

# 二十三、文件路径标识

推荐：

````markdown

```nginx

# /etc/nginx/nginx.conf

```

````

或者：

````markdown

```html

<!-- index.html -->

```

````

增强真实项目感。

---

# 二十四、语气风格

保留：

* 口语化

* 新手友好

* 降低焦虑

* 真人指导感

保留类似：

* “这是正常现象”

* “如果打不开”

* “说明已经成功”

增强教程陪伴感。

---

# 二十五、最终目标

最终 Markdown 必须：

* 可直接复制

* 可直接渲染

* 可直接部署

* 无语法错误

* 无代码块错误

* 无 Markdown 错误

最终效果必须像：

* 官方开发文档

* 高级独立技术博客

* 产品级技术教程

* 专业开发者内容

核心目标：

# 像真正的技术文档工程师写的

# 而不是 AI 随机生成的 Markdown
